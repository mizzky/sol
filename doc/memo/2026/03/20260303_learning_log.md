## 学習ログ — 2026-03-03

### 今回取り組んだタスク
- チケット20: カート機能の統合寄りコンポーネントテストを追加（`Add` → `Get` → `Update` → `Remove` → `Clear` のフロー）。
- テーブル駆動テストで主要ケース（正常系、認証未提供、AddでのDBエラー、所有権違反）を実装。

### ユーザーが質問した内容
- モックを連続的に呼ぶテストの意味（UTとの違い、価値、限界）。
- AddされたデータがGetで取れることをモック検証で確認する意義はあるか。

### 躓いた点と解決策
- 問題: テストのモック期待名のタイプミス（`GetOrCreateForUser` vs `GetOrCreateCartForUser`）でテストがpanic。
  - 対処: モック設定をハンドラの実際のメソッド名に修正。スタックトレースを参照して迅速に特定。
- 問題: テストケースで認証をセットするフラグを忘れ、期待されるHTTPステータスが `401` になった。
  - 対処: テーブル定義に `withAuth: true` を追加し、テストごとにコンテキストへ `userID` を適切に挿入。

### 学んだこと / 意見
- モック中心の「コンポーネントテスト」はルーティング・ミドルウェア・シリアライズ・エラー伝播の結合を高速に検証でき、開発ループに有効。
- 真の統合性（SQL挙動・トランザクション・マイグレーション不整合）はテスト用DBを用いる統合テストでしか検出できない。
- コスト対効果としては、まずはモック/フェイクで素早くカバレッジを上げ、本番に近い統合テストをCIで限定実行するのが現実的。
