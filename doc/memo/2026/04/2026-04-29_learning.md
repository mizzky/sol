## セッション 1 (15:00)

### 取り組んだタスク
- Issue #74 (ログ設計ドキュメント) の第1章「出力スキーマ」をユーザーと対話で設計
- 必須キーと optional キーの選定・役割整理（timestamp, level, event, message, error_type, status, method, route, request_id, user_id, trace_id, operation, client_ip を必須とし、path, duration_ms, cause を optional と決定）
- `operation` のスキーマ方針を「キーは必須、値は InternalError 以外は null を許容」と明文化
- `status` と `error_type` の役割分離（原因分類 vs 運用分類）を深掘りし、設計上の結論をまとめた
- `route` と `path` の使い分け（集計軸 vs 個別リソース特定）を整理

### ユーザーが質問した内容
- `error_type` と `status` の違いと、status を必須にする本質的理由
- `route` と `path` の違いと、どちらを必須にするか
- `request_id` / `user_id` / `trace_id` の役割の違いと trace_id の取り扱い
- `operation` を全エラーで持たせるべきか（今は InternalError のみ）
- `duration_ms` と `cause` の扱い（optional にするかどうか）

### 躓いたポイントと解決策
- 躓き: `status` が `error_type` と重複して見える点
  - 解決: 役割を明確化（`error_type`=原因、`status`=運用/監視の分類）し、両軸でログを残す方針に決定
- 躓き: `operation` を全エラーで持たせることへの違和感
  - 解決: スキーマではキーを常に出すが、値は InternalError でのみ必ず入れ、それ以外は `null` を許容する運用に決定
- 躓き: `path` を集計キーに含めるとノイズになるか
  - 解決: `route` を必須（集計軸）、`path` を optional（個別障害の追跡用）とする
