### 躓いたポイントと解決策

#### 1. 認証未認可時の user_id 出力方針の判断
**問題**: 
- UnauthorizedError(401) で user_id を出力する必要があるか？
- NotFound で統一した場合、原因切り分けが難しくなるか？

**検討・判断プロセス**:
- UT の目的は「ErrorHandler の slog 出力仕様」の検証であり、認証ロジック自体ではない
- 可読性と原因切り分けを優先→ **すべてのエラー時に user_id を省略（NotFound で統一）**
- 理由：未認証のリクエストでは user_id が確定できない状態であり、これはデザイン上の仕様として定義される

**実装判断**:
- UT は NotFound ケースにフォーカス
- user_id が **存在しない（0/null ではなく キー欠落）** ことを確認する検証方法を確立

#### 2. JSON decode 時の float64 化問題
**躓き**: UT で slog の JSON 出力をパースした際、user_id が int ではなく **float64** になった
**原因**: `json.Unmarshal()` は numeric value を型指定なしで float64 にデコード
**解決**:
```go
// 間違い: user_id を int で直接比較
var logData map[string]interface{}
json.Unmarshal(logBytes, &logData)
// logData["user_id"] は float64 型

// 正解: float64 で受け取り、型アサーションで検証
userID, ok := logData["user_id"].(float64)
if !ok {
  // キーが存在しない、または型が異なる
}
```

#### 3. UT で gin.Context の値を直接参照しない設計
**問題**: UT 本体から gin.Context の値を直接読み取り、ログ出力と比較する方法は脆弱性がある
**改善**: 
- **既知の Context 値**（例：user_id = 123）とログ JSON の内容を比較する方針に統一
- gin.Context の内部状態に依存しない
- ログが正しく出力されているかを黒箱検証

#### 4. slog attrs の可変長 append パターン
**躓き**: 認証済み時のみ user_id を attrs に追加したい場合、attrs を事前定義できない
**解決**:
```go
// 初期化：共通必須項目で作成
attrs := []slog.Attr{
  slog.String("request_id", ctx.GetString("request_id")),
  // user_id は未認証の場合は append しない
}

// 認証済みの場合のみ追加
if userID, ok := ctx.Get("user_id"); ok {
  attrs = append(attrs, slog.Int64("user_id", userID.(int64)))
}

h.log.ErrorContext(ctx, msg, attrs...)
```

- DEBUG レベルログの本番環境での扱いは？
- PII（個人情報）をログに出力する場合の方針は？
- エラーハンドリングの ErrorHandler 集約と通常イベントログの分離方針に矛盾はないか？
- ログ設計における「決めるべきこと」と「実装時に決めるべきこと」の線引きは？

### 躓いたポイントと解決策

#### 1. 「P2」と「PII」の用語確認
**躓き**: ユーザーが設計メモ中の「P2」という記述に対し、意図の確認が必要だった
**解決**: 
- 「P2」は表記揺れ。正確には「PII（Personally Identifiable Information）= 個人識別情報」
- 本来の意図：「ユーザーのメールアドレスなど、個人情報をログに出力してはいけない」
- 最終形：「通常イベントログでは PII 出力禁止。マスキング機能は最終防衛ではなく、最初から出さない設計」

#### 2. ErrorHandler 集約とイベントログの分離方針の矛盾検証
**躓き**: 
- エラーは ErrorHandler に集約する
- 通常イベントは各処理の終端で出力する
- この2つの方針が矛盾していないか不確実だった

**解決**:
- **矛盾していない**。役割が異なる：
  - ErrorHandler：エラー状態のみをキャッチ・集約・出力（同期・非同期を含む）
  - 通常イベントログ：リクエスト処理の成功パスにおいて「何が起きたのか」を記録
- エラーと成功を別軸で管理することで、ログの責任が明確化される

#### 3. 実装詳細をドキュメントのどこに記載するかの判断
**躓き**: 
- ログレベル定義（INFO / DEBUG の条件）は決まった
- しかし「ログ出力用のヘルパー関数をどう分担するか」「具体的なサンプルコード」をどのドキュメントに書くべきか不明確だった

**解決**:
- **ログレベル定義書（adr-structured-logging-error-policy-2026-05-14.md）**: アーキテクチャ判断のみ
  - INFO / DEBUG の出力条件、必須フィールド、命名規則など
  - 「何を記録するか」のポリシー定義に留める
- **ログ設計書（実装時に策定）**: 実装パターンとサンプルコード
  - ヘルパー関数の実装、具体的な logger インタフェース、事例コード
- 分担基準：**ポリシー（誰でも読める）vs 実装（エンジニア向け詳細説明）**

### 合意した設計方針（Ticket F 実装基準）

#### 1. INFO レベルの必須フィールド
```
- request_id
- method（HTTP メソッド）
- route（APIパス）
- status（HTTP ステータス）
- duration_ms（処理時間ミリ秒）
- event（イベント識別子）
- user_id（認証済みの場合のみ）
```

#### 2. event 命名規則
- **形式**: `lower_snake_case` の完了形
- **パターン**: `{domain}_{action}_{result}`
  - 例：`user_login_success`, `product_list_fetched`, `order_created_with_items`
  - 「~された」「~完了」の完了形で統一
- **命名時の留意**：処理内容を明確に表すが、冗長にならないこと

#### 3. ログ出力のルール
- **原則**: 1 リクエスト = 1 INFO ログ
- **例外**: 複数の状態遷移が発生するエッジケースのみ、追加で 1 件まで出力可能
  - 例：カートに商品を追加 → 在庫確認 → キャンペーン確認 → 販売可能判定→ ここまでで 1 件
  - その後、同一リクエスト内で「キャンペーン適用」という独立した状態遷移が発生した場合のみ、追加 1 件

#### 4. PII 出力ポリシー
- **原則**: ログに PII（メールアドレス、電話番号、名前など）を出力しない
- **マスキング方針**: マスキング機能は「最終防衛」ではなく、**最初から出さない設計**
- **例外**: ユーザー ID（純粋な ID のみ）は個人情報ではないため出力可能
- **実装責任**: ログ出力時に「本当に出力してよいか」の判定を組み込む

#### 5. DEBUG レベルの位置づけ
- **対象環境**: 開発環境が主
- **本番環境**: 原則無効
- **調査時**: 本番で一時的に有効化する場合は、調査期間中のみ。終了後は必ず無効化
- **出力内容**: 中間値、ルーティング情報、DB クエリなど「推移診断用」情報
- **ログ量**: 本番で常時有効化すると I/O 負荷が増加するため、慎重に運用

### 躓いたポイントと解決策

#### 1. ヘルパー配置の判断
**問題**: ヘルパーを middleware として実装すべきか、pkg 配下の汎用ヘルパーとすべきか不明確
**解決**:
- middleware は「HTTP リクエスト周期に紐付く」機能に限定
- ヘルパーは「各ハンドラが任意のタイミングで呼び出せる」汎用関数
- **結論**: ヘルパーは `backend/pkg` 配下に配置。middleware は request_id/duration_ms 基盤のみ担当

#### 2. request_id と duration_ms の「優先順」の誤解
**躓き**: 「request_id と duration_ms のどちらを先に確定するか」という優先順の質問があった
**解決**:
- これは「優先順」ではなく「**並行確定**」の問題
- 両者とも最上流（middleware）で同一タイミングで確定すべき
  - リクエスト受信時に request_id を生成 → Context に保存
  - 同時に start_time をContext に保存 → ハンドラ終端で duration_ms = now - start_time で計算
- **結論**: 競合ではなく、最上流で両方確定するのが最適

#### 3. 正常系ログの検証粒度
**問題**: 正常系ログが増えると、テストコード量が急増する可能性
**解決**:
- 検証対象を絞る：
  - ヘルパー自体のUT → 共通フィールド付与が正しいか確認
  - 全体UT → エンドツーエンドでイベント出力されているか確認
  - 各ハンドラUT → ハンドラ固有ロジックの検証（ログ出力は UT 対象外）
- **結論**: ハンドラUT では正常系ログ検証は過剰になるため、ヘルパーUT中心で進める
