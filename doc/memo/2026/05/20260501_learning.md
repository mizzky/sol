## セッション 1 (TDD Mentor Mode - ログ設計書深掘り)

### 取り組んだタスク

1. **Issue #74（ログ設計書）の読み込みと理解**
   - 出力スキーマの必須キー・optionalキー定義の確認
   - event命名規則（原則・フォーマット・NG/OK例）の学習

2. **構造化ログの設計思想の深掘り**
   - 検索性：message全文検索に依存せず絞り込みできること
   - 一貫性：同じ意味に同じキー名で表記揺れをなくすこと
   - 低カーディナリティ：eventの種類を増やしすぎない
   - 文脈連結：request_id/trace_id/user_idで点ではなく線で追跡
   - 機密保護：PIIや秘密情報を出さない

3. **event / message / error_type の役割分担の明確化**
   - error_type：エラーの型分類（粗い粒度、7種固定）
   - event：ユースケース単位の分類（中粒度）
   - message：人間向けの詳細記述（細かい粒度）
   - アラート・集計→error_type、機能特定→event、原因追跡→message

4. **statusとerror_typeの関係性の調査**
   - 当初の疑問：statusは必要か？
   - 調査結果：error_typeは内部分類、statusは運用分類（異なる責務）
   - 結論：運用監視・アラートルール策定でHTTPステータスは標準軸

5. **命名フォーマット決定**
   - {リソース名}_{動詞}_{結果} の snake_case で統一
   - 例：user_login_failed, product_fetch_not_found, cart_item_add_conflict

### 躓いたポイントと解決策

#### 躓き1：最初のアプローチ選択ミス
- **原因**：slogのAPI詳細から学習しようとしていた（ボトムアップ）
- **解決策**：TDD Mentorの指導により、設計思想を先に深掘りする方がより効果的であることに気づいた
- **学習点**：実装前に設計軸を理解することの重要性

#### 躓き2：statusの必要性への疑問
- **原因**：error_typeがあればステータスは不要ではないかという錯誤
- **解決策**：公式ドキュメント（semantic conventions、Go標準）を参考に、error_typeとstatusが独立した責務を持つことを理解。error_typeは内部分類（開発者視点）、statusは運用分類（監視・アラート視点）と相互補完的
- **学習点**：同じ概念に見えても異なる責務を持つ設計の重要性
