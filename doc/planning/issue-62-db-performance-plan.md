# Issue #62 大量データ下のEC主要クエリ最適化・実行計画検証

## 1. 目的

性能検証専用DB `coffeesys_perf` に再現可能な大量データを投入し、ECの主要クエリを `EXPLAIN ANALYZE` とアプリケーション側の計測結果に基づいて評価する。

対象は商品一覧、ユーザー別注文一覧、注文明細、カート取得、N+1比較、DB接続プールである。推測で最適化せず、before/afterの実行計画と計測値を根拠に変更の採否を判断する。

## 2. スコープと固定方針

- PostgreSQL 17、Gin、`database/sql`、sqlcの現行構成を維持する。
- 性能検証には開発DBと分離した `coffeesys_perf` を使用する。
- 性能DB内の業務データは、DB名を検証した後で全削除可能とする。
- OFFSET/keyset paginationの比較は検証専用とし、本番APIやフロントエンドには導入しない。
- N+1比較コードは本番処理と分離する。
- GitHub Pages向け成果物は `doc/` 配下のMarkdownまでとし、公開設定は対象外とする。
- Issue #62の各ミニタスクは、カスタムCodexサブエージェント `tdd_mentor` を起動し、その段階的メンタリングをベースに進める。
- 1ミニタスクを30〜60分、原則1コミットで進める。
- small、medium、largeの順に検証し、前段階が完了するまで次の規模へ進まない。
- 最初に着手するタスクは **0-1 性能DBの接続確認** とする。

##### 共通の決定的生成規則

- IDは全テーブルで1始まりの連番を明示投入し、投入後に各sequenceを最大IDへ同期する。
- 乱数と`NOW()`は使わず、固定基準時刻、ID、整数除算、剰余だけから値を生成する。
- usersは原則`member`かつ`active`とし、password hashには性能検証専用の固定された非本番値を使う。
- 固定名と一意値は次の形式とする。
  - category: `category-01`から`category-10`
  - product name: `product-%07d`
  - SKU: `PERF-%07d`
  - user name: `user-%06d`
  - email: `user-%06d@example.test`
- `.test`はテスト用の予約TLDだが、誤送信時のDNS失敗・再試行を避けるため、メール送信機構の無効化は別の環境安全策として扱う。
- 商品価格は`100 + ((product_id - 1) % 1000) * 10`とし、100円から10,090円までを10円刻みで循環させる。

##### 注文集中、OFFSET、ページ境界

- `user_id = 1`を集中ユーザーとし、全注文の20%を割り当てる。

| Profile | 集中ユーザーの注文数 | 主に検証する深いOFFSET |
|---|---:|---:|
| small | 2,000 | 1,000 |
| medium | 20,000 | 10,000 |
| large | 200,000 | 100,000 |

- `order_id = 1`から集中件数までを`user_id = 1`へ割り当て、その後の80%は`user_id = 2`以降へround-robinで割り当てる。集中ユーザーの最新60注文は、集中範囲の末尾60 IDとする。
- largeでは同じデータセットに対して`OFFSET 0 / 1,000 / 10,000 / 100,000`、`LIMIT 50`を計測する。集中率ごとにDBを再生成せず、OFFSETだけを変えて比較条件を揃える。
- 通常注文の`created_at`は`2025-01-01 00:00:00+00`へ`order_id`秒を加え、`updated_at = created_at`とする。
- 集中ユーザーの最新60注文だけは、全注文の通常最大時刻より後となる同一の固定時刻へ上書きする。実装時は、例えば基準時刻へ`orders総数 + 1`秒を加えた値を使う。
- 同一時刻60件は、`LIMIT 50`の1ページ目と2ページ目をまたがせるための正しさの検証データである。`created_at`だけのcursorでは残り10件が欠落し得ること、`(created_at, id)`なら欠落しないことを確認する。
- 最新60件の境界検証と、100,000行を読み飛ばすOFFSET性能測定は目的を分離する。

##### 注文statusと注文明細

- ordersのstatusは`pending` 90%、`cancelled` 10%とし、`order_id % 10 = 0`だけを`cancelled`にする。
- `cancelled`は`cancelled_at = created_at + 1 day`、`pending`は`cancelled_at = NULL`とする。
- 注文明細数は次の分布とし、全profileで平均3明細と既定の総件数を維持する。

| Profile | 1明細の注文（20%） | 2明細の注文（50%） | 6明細の注文（30%） | order_items合計 |
|---|---:|---:|---:|---:|
| small | 2,000 | 5,000 | 3,000 | 30,000 |
| medium | 20,000 | 50,000 | 30,000 | 300,000 |
| large | 200,000 | 500,000 | 300,000 | 3,000,000 |

- statusと明細数の不要な相関を避けるため、`item_bucket = floor((order_id - 1) / 10) % 10`とし、bucket 0〜1を1明細、2〜6を2明細、7〜9を6明細にする。各10注文のまとまりにstatus 10%が含まれるため、各明細数でもstatus比を維持できる。
- 各明細の1始まりの番号を`item_ordinal`として、product_idは`(((order_id - 1) * 6 + item_ordinal - 1) % products総数) + 1`で割り当てる。同一注文は最大6明細なので重複しない。quantityは`1 + ((order_id + item_ordinal - 2) % 3)`で1〜3を循環させる。
- `unit_price`と商品snapshotは参照商品の値から生成し、`orders.total = SUM(order_items.quantity * order_items.unit_price)`を必ず満たす。

##### 商品とカテゴリ

- `is_available = true`を95%とし、`product_id % 20 = 0`だけをfalseにする。
- stockは0個10%、1〜5個20%、10〜100個60%、500〜1,000個10%とする。
- `stock_bucket = floor((product_id - 1) / 20) % 10`とし、bucket 0を0個、1〜2を1〜5個、3〜8を10〜100個、9を500〜1,000個に割り当てる。範囲内の値はproduct_idの剰余で決める。
- 具体的には、低在庫を`1 + ((product_id - 1) % 5)`、通常在庫を`10 + ((product_id - 1) % 91)`、大量在庫を`500 + ((product_id - 1) % 501)`で生成できる。
- 販売可否と在庫は完全には連動させず、`availableかつstock = 0`の売り切れ商品と、`unavailableかつstock > 0`の販売停止・残在庫商品を含める。20商品単位の各stock bucketへfalse商品が1件ずつ入るため、この組み合わせを決定的に作れる。
- この比率は実運用統計ではなく、偏りと境界値を作るための学習用仮定である。現行の全件商品一覧では主要な性能変数として扱わない。
- categoriesは全profileで10件とする。`r100 = (product_id - 1) % 100`により、0〜39はcategory 1、40〜59はcategory 2、60〜99は`3 + floor((r100 - 60) / 5)`へ割り当て、category 3〜10を各5%にする。

##### cartsとcart_items

- user IDの前半50%に各1cartを作り、1ユーザーあたり最大1cartとする。
- cart 1は空、cart 2は100明細、残りは各3明細とする。期待件数は`3 * (carts - 2) + 100`である。
- 同一cart内でproduct_idを重複させず、quantityは必ず1以上、すべての外部キーを有効にする。
- `cart_items.price`は現行handlerの扱いに合わせて参照商品の単価と一致させる。
- 通常・大量cartの商品は、`is_available = true AND stock_quantity > 0`の商品をproduct_id順に並べた候補集合から、cart_idと1始まりの`item_ordinal`を使った剰余で決定的に割り当てる。1cartの最大100明細内で候補が重複しない連続位置を選ぶ。

##### 再現性と整合性の期待値

- 同じprofileを安全に初期化して2回seedし、件数、分布、代表fingerprintが一致し、データが増殖しないことを確認する。fingerprintには各表の`count / min(id) / max(id)`と、安定した数値列の`sum`などを使う。
- 初期化対象は業務テーブルだけとし、`schema_migrations`を含めない。
- profileごとの検証SQLで、少なくとも次を確認する。
  - 全テーブルの件数がprofile表と一致する。
  - 集中ユーザーの注文が全体の20%、statusが90%/10%、集中ユーザーの最新同時刻注文が60件である。
  - 商品利用可否、stock bucket、カテゴリの件数が期待比率と一致する。
  - email、category名、SKU、`(cart_id, product_id)`に重複がない。
  - 全外部キー関係の孤児レコードが0件で、order_itemsとcart_itemsのquantityが1以上である。
  - 注文ごとの明細数が1、2、6のいずれかで、その注文数と総明細数が期待値に一致する。
  - 注文合計と明細合計の不一致が0件である。
  - cartの明細数が0、3、100のいずれかで、各パターン数とcart_items総数が期待値に一致する。
  - `cancelled`だけがcancelled_atを持ち、`pending`のcancelled_atがNULLである。

## 5. TDDとコミット方針

実装を伴うミニタスクは、次の順で進める。

1. スコープと完了条件を再確認する。
2. 整合性条件または期待動作をテストとして先に表現する。
3. Redを確認する。
4. 最小実装でGreenにする。
5. 重複や責務を整理する。
6. 関連テストを再実行する。
7. 引き継ぎジャーナルを更新してコミットする。

推奨コミット単位は次のとおり。

1. `test(perf): define seed validation queries`
2. `feat(perf): add reproducible performance dataset`
3. `docs(perf): record baseline query plans`
4. `perf(db): refine order index and remove redundant indexes`
5. `test(perf): compare n-plus-one and batch loading`
6. `feat(db): configure connection pool`
7. `docs(perf): publish issue 62 findings`

## 6. 計測ルール

- `EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)`を使用する。
- データ件数、実行SQL、パラメータ、インデックス、PostgreSQLバージョンを併記する。
- ウォームアップ後5回実行し、中央値を代表値とする。
- OSキャッシュを強制削除したコールド計測は対象外とする。
- 実行時間だけでなくscan type、cost、actual time、rows、loops、buffersを保存する。
- 実行時間の差を自動テストのPASS条件にしない。

## 7. Definition of Done

- 3つ以上の主要クエリについてbefore/afterの実行計画がある。
- インデックス変更または変更しなかった判断を実行計画に基づいて説明できる。
- 複合インデックスのカラム順序を説明できる。
- N+1版とbatch版をクエリ回数と実行時間で比較している。
- OFFSETとkeyset paginationの差を説明できる。
- `sql.DB`接続プール設定の採用理由を説明できる。
- DB設計ガイドラインが `doc/` のMarkdownに整理されている。
- `go test ./... -count=1`がPASSする。
