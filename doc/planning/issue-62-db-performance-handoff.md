## 4. 固定済みの判断

- 性能検証専用DBは `coffeesys_perf` とする。
- 性能DB内の業務データは、DB名確認後に全削除可能とする。
- productsとordersはlargeでそれぞれ1,000,000件とする。
- order_itemsはlargeで3,000,000件とする。
- ページネーションは検証専用とし、本番APIやフロントエンドを変更しない。
- GitHub Pages向け成果物は `doc/` 配下のMarkdownまでとする。
- 段階的メンタリングで進め、ユーザーが学習しながら実装する。
- 1タスク30〜60分、原則1コミットとする。
- small、medium、largeの順で進める。
- 最初に実施するタスクは **0-1 性能DBの接続確認** とする。

## 5. 現在把握しているリポジトリ状況

- 開発環境はWSL2上のVS Code Dev Containerである。
- フロントエンドはNext.js、バックエンドはGinである。
- DBアクセスはPostgreSQL 17、`database/sql`、sqlcを使用している。
- migrationはv12まで存在する。
- `orders`には現在 `(user_id, created_at DESC)` のインデックスがある。
- `order_items`には `order_id` と `product_id` のインデックスがある。
- `ListProducts`は`ORDER BY id`による全件取得で、ページネーションはない。
- `ListOrdersByUser`は`WHERE user_id = $1 ORDER BY created_at DESC`で、ページネーションはない。
- `ListOrderItemsByOrderID`は`order_id`で検索する。
- カート取得はcarts、cart_items、productsをJOINしており、既にN+1を回避している。
- `main.go`は `DATABASE_URL` を読み取るが、接続プール設定は追加されていない。
- UNIQUE制約由来のインデックスとの重複候補がある。
  - `products_sku_key` と `idx_products_sku`
  - `carts_user_id_key` と `idx_carts_user_id`
  - `cart_items_cart_id_product_id_key` と `idx_cart_items_cart_id`
- 重複候補は推測で削除せず、利用状況とサイズを計測して判断する。

## 6. ユーザーから報告された進捗

- `coffeesys_perf`は作成済み。
- performance DBへmigrationを適用済み。
- `.env`へ `PERF_DATABASE_URL`を設定する方針が決まっている。
- `.vscode/tasks.json`に性能検証DBを利用するバックエンド起動タスクを追加済み。
- アプリケーションは従来どおり `DATABASE_URL`のみを読み、Task起動時に `PERF_DATABASE_URL`を `DATABASE_URL`として渡す。
- 通常バックエンドと性能検証バックエンドは同じ8080番ポートを使うため、同時起動しない。

上記はユーザー報告に基づく。実際の接続先、migration version、現在件数はタスク0-1で再確認する。

## 9. 安全規則

- 接続文字列、パスワード、トークンをログやMarkdownへ記録しない。
- destructiveなSQLの前に `current_database()` を検証する。
- `coffeesys_perf`以外ではseed/resetを即時中止する。
- `schema_migrations`は初期化対象に含めない。
- large投入前にsmallとmediumを完了させる。
- 実測前に `ANALYZE` を実行する。
- 実行時間だけでなくscan type、cost、actual time、rows、loops、buffersを保存する。
- 性能差を自動テストの合否条件にしない。
- 通常バックエンドと性能検証バックエンドを同時に起動しない。
- ライブラリの追加や開発環境を変更するコマンドは、実行前にユーザーへ説明して確認する。
