# GitHub Copilot モード活用

## 1. GitHub Copilot 各モードの使い分け（写経・ドキュメント自動生成派向け）

実装を自分で行い、ドキュメント生成や方針相談にCopilotを活用する場合の最適フロー：

| モード | 推奨される活用シーン | 特徴 |
| :--- | :--- | :--- |
| **Plan** | **メインの相談・設計フェーズ** | 循環参照の解決策など、文脈を維持したまま深掘りする。`instructions.md` を遵守し、方針の合意形成を行う。 |
| **Agent** | **自動化・事務作業フェーズ** | Planで決定した内容に基づき、ドキュメント（README/docs）を自動更新させる。 |
| **Ask** | **リサーチ・横断調査フェーズ** | 今の実装プランを汚さずに、広域なコード調査や外部ライブラリの仕様、思考実験（もし〜なら？）を行う。 |
| **Edit** | **局所的な修正** | 数行のリファクタリングなど。写経派の場合は使用頻度は低い。 |

## 1. 効率的な運用のコツ
* **Planモードでの深掘り**: 実装中の疑問はPlanモードのまま投げたほうが、現在の提案コードという「文脈」を共有できるため精度が高い。
* **ガードレールの活用**: `instructions.md` で編集禁止ディレクトリを設定している場合、Planモードが勝手にファイルを書き換える心配がないため、安心して議論に集中できる。
* **使い分けの基準**: 「今の実装手順（Plan）に影響を与えたいか（Plan）」vs「純粋に知見だけ得たいか（Ask）」で判断する。
